## Catalog-pool blowup + memory ramp — 2026-07-22 (ID `PR#`, continues the perf/incident series)

**What happened.** Railway's memory graph ramped 500 MB → 2.0 GB overnight with nobody using the app. It is **not** a leak: `/api/health` (extended in `c670dd6`) showed the Node process flat at 354 MB RSS / 100 MB heap and Litestream at 58 MB, while `cgroup.fileMb` — kernel page cache — went 2 MB → 488 MB in 41 minutes. The container is caching pages of a **2.5 GB `rr.db`**. Reclaimable, will not OOM (limit 7629 MB); it is a **cost** problem whose fix is shrinking the file.

**Root cause** (`/api/dev/dbsize`, added in `7f8c4a7`): `media_items` **675,787 rows**, `media_links` 678,962, `media_external_ids` 678,965 — against **1 user**, 1,912 `user_library`, 96 watchlist, 2,337 `user_item_state`. The pool is ~353× the real library. `freelistCount: 0`, so **VACUUM alone reclaims nothing** — the file is full of real rows, not holes. The bytes are `media_links.raw_data` (a full source-JSON blob per row, ~3.5 KB each). Public facet pages are provider-sourced and persist **every rendered title** thin (`browsed=1`) via `persistDiscoverItems`, `FACET_PAGE_SIZE = 60` per page (`publicFacetDetail.ts:478`); those pages are public and crawled, so every crawler walk mints up to 60 rows. Unbounded by construction.

**Decisions locked (2026-07-22, user):** ① stop the growth by persisting **only for logged-in users** — anon/crawler views render non-linkable titles; ② **prune + VACUUM** the existing tail, accepting 404s on browsed item URLs; ③ **PR13–PR15 are safe for Sonnet; PR16 stays in a full-effort Opus session** per AGENTS.md's "don't delegate matcher write paths / the pool invariant" rule.

- **PR16 (superseded in-progress notes, kept for the diagnostic corrections)** 🔵 2026-07-22 — tooling shipped (`8b4831e`, `c067a91`, `d5879f5`, `fc32514`, `5b67894`); paced background job running against prod. **Two of my own diagnoses were wrong and are corrected here so nobody re-derives them:** (1) there is **no checkpoint stall** — `wal_autocheckpoint` is 1000 (Litestream does NOT disable it) and at rest the WAL held `logFrames 1 / checkpointedFrames 1 / pendingFrames 0`, i.e. a 130 MB file with one already-checkpointed frame. A SQLite WAL doesn't shrink on an ordinary checkpoint; it's reused in place at its high-water mark, which is why `walMb` looked frozen at 60.7 for 12 h. Also: `wal_checkpoint(TRUNCATE)` is not a usable lever, it returns `busy=1` because Litestream holds a read lock over unshipped frames — that lock is what makes the backup trustworthy. **What actually paces the job is SQLITE_BUSY**, not the free-space floor (which never fired once): each retry is a 2–20 s pause, so the job self-throttles against Litestream's real capacity. Treating BUSY as backpressure rather than fatal is what made it viable — the first run died after 4,000 rows on one lock. Preconditions all verified from inside the container: Litestream snapshot `18d8221abccc198d` (1.08 GB, 2026-07-21T20:35Z) + live WAL segments, `df` 4.6 G / 2.5 G used / 2.1 G free. ⚠️ **Railway volume backups are Pro-plan-only and show "No Backups" — Litestream is the ONLY recovery path.** Predicate is wider than planned (excludes `user_item_state` **and** `user_library` **and** `user_watchlist`, since the latter two also cascade). Running tally: user rows untouched on every single pass (`user_library` 1912, `user_watchlist` 96), zero orphans throughout. **Still to do: finish the drain, then VACUUM** (blocked until `liveDataMb` drops — the guard measures live pages, not file size, deliberately).

### Production incident — 2026-07-21 (memory ramp returns: the image optimizer, ID `PR#` continued)
Memory ramped again the day AFTER the PR1–PR9 deploy — 7.52 GB RSS at 09:00, with the Railway error log now **entirely** `upstream image response timed out/failed`. The PR3 heap cap was working and irrelevant: `--max-old-space-size=1536` bounds the V8 old space, and ~6 GB of that RSS was **native** memory (sharp/libvips + allocator), which it does not touch.

**Root cause: `/_next/image` was decoding full-size RAWG originals in-process, dozens of times per page view.** Two multipliers stack:

1. **The sources are large in PIXELS.** RAWG poster URLs are persisted as originals (`normalize.ts` → `d.background_image`, no resize segment — **726/726 rows**). Measured over a random 25-poster sample: median **1920×1080 → 5.9 MB decoded**, worst seen **4161×2607 → 31 MB decoded**. **File size is a poor proxy** — one 219 KB file was 3840×2160 and decoded to **23.7 MB**. libvips allocates on pixel count, not bytes on the wire.
2. **Each image spawns one sharp job PER SRCSET WIDTH.** `sizes="64px"` (no `vw` unit) makes Next emit its entire `allSizes` list — **11 widths**. Measured on the live `/game/…/tarachine`: 5 distinct RAWG images → **55 distinct optimizer jobs for a single page view**, each decoding the full original. A 64 px thumbnail was doing 11 full-size decodes.

So one cold crawler hit on a game page ≈ **55 × ~6 MB ≈ 325 MB** of transient native allocation, with no concurrency limit in front of the optimizer, on a container capped at **0.4 vCPU** — and glibc then holding that memory in per-core arenas rather than returning it (PR11). That is the ramp.

**Correction to the first read of this incident:** the TMDB timeouts are *not* purely event-loop starvation. The 2026-07-21 log shows TMDB returning **502s** at 03:35, so at least one cluster was genuinely upstream-side. The memory diagnosis doesn't rest on them — it rests on the decode math above. TMDB/Steam/IGDB sources are far smaller in pixels and were collateral, not cause. Note also that **no RAWG line appears in the error log at all**: RAWG's CDN was healthy, so its (expensive, successful) work was invisible there. Absence from the error log is not absence of load.

- **PR10** ✅ 2026-07-21 — **Images bypass the optimizer entirely; the CDNs resize instead.** New `src/lib/imageLoader.ts` wired as `images.loader:"custom"` + `loaderFile`: it rewrites each URL to that CDN's own size variant — TMDB `w`-buckets, RAWG `/media/resize/{w}/-/`, IGDB `t_` tokens, Steam/unknown passthrough — and the browser fetches it directly. No upstream buffering, no sharp, no optimizer disk cache, and **image bytes stop transiting Railway egress**. Every bucket value was curl-verified against the live CDN before being hard-coded; 13 unit tests in `imageLoader.test.ts` pin the mapping (incl. no double-transform on already-sized RAWG URLs, and no rewriting of lookalike hosts). **TMDB rounds DOWN, RAWG rounds UP** — deliberate asymmetry, see the comments: TMDB's alternative is a smaller variant of a heavy JPEG, RAWG's alternative is the multi-MB original.
  - **Accepted tradeoff (Nils's call, 2026-07-21):** movie/show posters go from ~26 KB WebP to ~65 KB JPEG (measured, 5-poster sample, old prod `/_next/image?w=384` vs new `w342`) — a Discover page of 48 posters goes ~1.25 MB → ~3.1 MB. Game art gets dramatically lighter. Chosen over a RAWG-only hybrid because it removes the failure mode rather than shrinking it, and crawlers (the dominant traffic) gain nothing from WebP. Images now come from global CDNs and stay lazy-loaded. **If page weight ever needs winning back, the lever is Cloudflare (Polish/Resizing), not re-enabling sharp on a 0.4-vCPU box.**
  - `MediaGallery.tsx`'s hero is a raw `<img>` (deliberately — see its comment) and so was shipping the **full-size RAWG original straight to browsers** (up to 4 MB / 4161×2607); now routed through `cdnImageUrl(src, 1280)`.
  - **Left alone deliberately:** `sizes="64px"` on the thumbnail strip still emits 11 srcset entries. That was pathological only because each entry was a server-side decode — with the loader it costs nothing (the browser picks one CDN URL). Not worth churning the components for.
  - `next.config.ts`: `remotePatterns` kept as the documented host allowlist (Next no longer enforces it once nothing reaches the optimizer); `minimumCacheTTL`/`formats`/`qualities` marked inert-but-kept as the tuned fallback. `deviceSizes` is still live — it decides which widths the custom loader is asked for.
- **PR11** ✅ 2026-07-21 — `MALLOC_ARENA_MAX=2` (Dockerfile). glibc opens up to 8 malloc arenas **per core**, sized off the host's core count rather than the container's CPU limit, and freed native memory sits in those arenas instead of returning to the OS — RSS ratchets up and never comes back down, which is exactly the shape of this ramp. Applies to every native allocator in the process (better-sqlite3, zlib, TLS).
- **PR12** ✅ 2026-07-21 — `/api/health` now reports `memoryMb` (`rss`/`heapTotal`/`heapUsed`/`external`/`arrayBuffers`). Railway only graphs total RSS, which cost **two incidents' worth of guessing about where the memory was**. Rule of thumb for next time: *if `rss` ramps while `heapUsed` stays flat, it is not a JS leak — stop reading cache code.*

**Two log lines PR10–PR12 do NOT explain** (2026-07-21, one occurrence each, flagged not fixed): `TypeError: fetch failed`/`ETIMEDOUT` at 05:55 and a bare `TimeoutError` (code 23, an `AbortSignal.timeout`) at 08:40. Both look like a `httpFetch` (`http.ts`, 20 s default) escaping a caller that doesn't try/catch — that file's header claims "each adapter/sync step already try/catches", which these suggest isn't universally true.
